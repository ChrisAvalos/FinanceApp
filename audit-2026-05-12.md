## Still-present issues (calibration for "not 100/100")

### 🔴 Real bugs — high impact

1. **FIRE projection panel is broken when net worth is negative.** Frontend posts `starting_cents=-106554` to the API, which has a Pydantic `ge=0` validator. Response is a raw `422 Unprocessable Content — [{"type":"greater_than_equal", ...}]` blob, displayed verbatim to the user where the Monte Carlo chart should be. Drops the panel from 94 → 59. **Fix:** clamp negative starting balances to 0 server-side and render a one-line note ("Your tracked net worth is currently negative; FIRE projection starts from $0"); OR relax the validator to allow negative values and adjust the simulator.

2. **Net worth header shows wrong ASSETS total.** Header card says ASSETS $168.94 but the breakdown immediately below sums to $916.16 (the accounts genuinely total $916.16: Chase $463.25 + E*TRADE Savings $400.43 + E*TRADE Brokerage $52.48 + Albert $0 + E*TRADE Stock Plan $0). Drops trust score for the panel. **Fix:** trace which aggregation feeds the header card; the breakdown query is the canonical one.

### 🟠 UX warts — slow, not broken

3. **Trends mid-month over-correction (Sprint 25 follow-up).** Pro-rating the current month fixes the "-100% trailing avg" alarm but over-inflates trends for categories with sparse current-month data. E.g. Clothing shows **+1700%** and Travel **+2854%** because day 12 of the month projects up by ~2.58×. Should add a minimum-current-month-sample threshold before applying pro-rate (e.g. only pro-rate if current-month txn count ≥ 3 in that category).

4. **Redress matcher claims "Connect Plaid to get coverage" when Plaid IS connected.** "No catalog companies matched your last 2 years of transactions. Connect Plaid to get coverage — without transactions, the matcher has nothing to compare your spend history against." But the user has 3 Plaid items and ~387 outflow transactions in the 90-day window. The matcher just hasn't run, or hasn't matched anyone. Copy should distinguish "no matches yet — run the matcher" from "no Plaid data at all". Marked **Never synced** chip top-right is the real signal but it's small.

5. **Card offers Amex auth missing.** "Auth missing" + bootstrap-command displayed (`python -m finance_app.scrapers.offers.bootstrap amex`). Chase Offers ready but pipeline shows `Never synced`. Hitting `Scrape now` would fix the empty list. Knocks the panel from 86 → 81 because the headline number is $0.

6. **Tax export "Categorized but not tax-mapped" includes $10K Transfer.** "Transfer $10,207.45" appears under the categorized-but-not-tax-mapped section. That's correct technically (transfers aren't deductible) but flagging $10K of money movement next to legitimate deductions can mislead at a glance. Either move Transfer/Credit Card Payment/Savings under a distinct "Money movement (not tax-relevant)" subsection, or filter them out entirely.

7. **Class actions stat cards still don't filter by state.** (Carry-over from prior audit.) Click California → pending-potential / quick-wins / triage numbers stay at global totals.

8. **window.confirm() callsites remain** in Goals, CanonicalProducts, Receipts, Deals, Unclaimed, CardApplications, LegalClaims. The hook + toast component already exist; each conversion is ~2 lines.

### 🟢 Architectural cleanups

9. **Subscriptions row #25 "Peacock Premium" still has notes `Monthly, renews 3rd — from iPhone`.** That's how it was manually added. After Sprint 23c's catalog-aware dedupe, this row should pull a canonicalized display name + price from the catalog so it doesn't drift from future detector runs.

10. **Holdings preview is mock data.** The PREVIEW card showing $248,500 / +29.3% / 14 holdings is great design but the real Holdings table can only populate if Plaid grants the `investments` product. Either request investments at link time (some institutions deny it), or surface a clearer "ask your bank to enable investments" CTA. Current empty state's "Open Bank connections →" is correct but the path forward from there is unclear.

11. **Cross-store deals can't fire without scrapers and shopping patterns.** Walmart ready, Target / Costco / Amazon Fresh / Kroger all "needs auth". Until at least one is bootstrapped + receipts are logged + Detect runs, the panel sits empty by design. Honest, but the dollar-headline (Active deals 0, Annual savings $0) makes the panel feel weak.

### Why we're at 88, not 100

The two big regressions (FIRE −35, Net Worth −8) account for most of the audit-day drag. Both are caused by negative net worth — a real life event that the app's projection / aggregation logic didn't anticipate. The fixes are small (one validator change, one aggregation trace) and would lift the overall score by ~1.5 points.

Beyond that, the persistent ceiling drivers are the same as the last audit:

- **Intelligence** (8.6/10): catalog freshness still a quarterly job we haven't fully automated; cross-account inference still missing (e.g. "your AAA renews annually" needs 2+ years of data to detect); no semantic LLM extraction of email *content* (we use regexes — though the LLM-Gmail discovery in Sprint 16 is the first real LLM-on-email surface).
- **Accessibility** (8.0/10): unchanged. No focused a11y pass since project start. Things like `aria-current` on the active sidebar item, screen-reader-only labels on icon-only buttons, and `prefers-reduced-motion` honoring would each add 0.5–1.0 point.
- **Delightfulness** (8.3/10): big lift today (Hi Chris 🤚 on multiple panels, preview banners, HSA compounding copy). Still no celebratory moments after major actions ("you cancelled Calm — saving $14.99/mo / $179.88/yr"), no animated transitions on data updates outside CountUp on FIRE.
